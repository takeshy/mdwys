# mdwys メモ機能(Document Timeline)仕様書

## 1. 概要

各ドキュメント(Markdown / PDF / EPUB / テキスト / HTML)に対して、Timeline 形式のメモファイルを 1 対 1 で関連付ける。ウィジェットのヘッダーのメモアイコンからタイムラインパネルを開き、ドキュメント内のハイライト箇所とメモエントリを双方向にリンクできるようにする。

本仕様は「何を実現するか」を定義する。内部実装(状態管理、コンポーネント分割等)は既存コードベースの流儀に従うこと。

**前提作業:** PDF 表示を現行のネイティブビューア(iframe 埋め込み)から pdf.js ベースの実装に移行する(§6.1)。現行実装ではテキストレイヤー・選択イベント・ページ制御が一切利用できず、PDF に対する本機能が成立しないため。

## 2. 用語

| 用語 | 意味 |
|---|---|
| メモディレクトリ | 全メモファイルを格納する、設定で指定された単一ディレクトリ |
| メモファイル | ドキュメント 1 つに対応する Timeline 形式の Markdown ファイル |
| エントリ(post) | メモファイル内の `---` 区切りの 1 単位 |
| アンカー | ドキュメント内の位置を指す情報。エントリに付与される |
| アンカー付きエントリ | source / アンカー情報を持つエントリ |

## 3. 設定

Settings 画面に以下を追加する。

- **メモディレクトリ**: ディレクトリパス。未設定時はメモ機能全体を無効とし、メモアイコンクリック時に設定を促すダイアログを表示する。
- 既存の設定永続化の仕組みに追加する。

## 4. メモファイルの命名規則

### 4.1 エンコード規則

ドキュメントの絶対パスを以下の規則で 1 つのファイル名にエンコードし、拡張子 `.md` を付与する。

変換は入力を 1 文字ずつ走査し、以下の順で置換する:

| 入力文字 | 出力 |
|---|---|
| `_` | `_u` |
| `/` | `_s` |
| `\` | `_s`(**Windows のみ**。POSIX では `\` は合法なファイル名文字であり、変換すると `/` を含むパスと衝突するため変換しない) |
| `:` (Windows ドライブレター等) | `_c` |
| その他 | そのまま |

デコードは `_` の直後の 1 文字(`u`/`s`/`c`)を見て逆変換する(`_s` は `/` に戻す)。`_` の後に `u`/`s`/`c` 以外が来るファイル名は不正として扱う(手動 rename ミスの検出)。

**例:**

- `/Users/takeshy/books/go_book.pdf` → `_sUsers_stakeshy_sbooks_sgo_ubook.pdf.md`
- `C:\Users\takeshy\doc.md` → `C_c_sUsers_stakeshy_sdoc.md.md`

**可逆性の担保:** 変換先(`_u`, `_s`, `_c`)は互いに素であり、`a_/b` → `a_u_sb`、`a/_b` → `a_s_ub` のように一意にデコード可能であること。ユニットテストで往復変換(encode → decode → 元と一致)を必須とする。

### 4.2 長さ制限フォールバック

エンコード後のファイル名が **200 バイト(UTF-8)** を超える場合のみ:

1. エンコード名を先頭から 180 バイト以内(UTF-8 文字境界で切る)に切り詰める
2. 元の絶対パスの SHA-256 ハッシュ先頭 8 文字(hex)を `.` 区切りで付与する
3. 例: `<切り詰めた名前>.a3f2c1d9.md`

フォールバック発動時、デコードによる元パス復元は不可となるため、ファイル先頭の frontmatter に `source:` として元の絶対パスを必ず記録する(§5.1)。

### 4.3 解決順序

ドキュメントからメモファイルを探す際:

1. 絶対パスに §4.1 と §4.2 を適用した**正規名**(200 バイト超の場合はフォールバック名)のファイルを探す
2. なければ、正規名で新規作成(初回投稿時に作成。空ファイルは作らない)

## 5. メモファイルのフォーマット

### 5.1 ファイル構造

```markdown
---
source: /Users/takeshy/books/go_book.pdf
---

<エントリ>
---
<エントリ>
---
<エントリ>
```

- frontmatter の `source` は元ドキュメントの絶対パス。フォールバック命名(§4.2)時の復元、およびドキュメント移動時の手動修正ポイントとして常に書く。
- エントリは **古い順**。新規投稿は常に**ファイル末尾へ追記**する(既存部分を書き換えない)。
- エントリ区切りは前後に空行を伴う `---`。

**パース時の frontmatter 処理(必須):** エントリ分割の**前に** frontmatter を除去する。ファイルの 1 行目が `---` の場合、次に現れる `---` のみの行までを frontmatter として読み、それ以降をエントリ本体とする。除去せずに `---` で分割すると frontmatter のフェンスが区切りと誤認され、`source:` 行が「本文のみのエントリ」としてタイムラインに表示されてしまう(gemihub の Timeline パーサをそのまま流用すると発生するため注意)。

**区切りの判定(必須):** エントリ分割は「**空行に挟まれた** `---` のみの行」だけを区切りとして扱う(gemihub の `split(/^\s*---\s*$/m)` より厳格)。本文中のコードブロック内の `---` や Markdown 水平線での誤分割を減らすため。

### 5.2 エントリのフォーマット

```
2026-07-02T10:15:00.000Z
id: 20260702-191500-000
anchor: page=12
quote-prefix: 直前の文脈テキスト
quote-suffix: 直後の文脈テキスト

> ハイライトした引用文

メモ本文(Markdown。wikiリンク・画像埋め込み・タグ可)
```

**フィールド定義:**

| 行 | 必須 | 内容 |
|---|---|---|
| 1 行目 | 必須 | UTC ISO 8601 タイムスタンプ(ミリ秒精度) |
| `id:` | 必須 | ローカル時刻ベース `YYYYMMDD-HHMMSS-mmm`。ファイル内で一意 |
| `anchor:` | 任意 | アンカー指定(§6)。無しなら通常のメモ |
| `quote-prefix:` / `quote-suffix:` | 任意 | 引用文の直前/直後 30 文字程度。同一文言が複数箇所ある場合の曖昧性解消用。取得時に空白正規化(§6)して**単一行**で格納する |
| `pinned:` | 任意 | `true` でピン留め(既存 Timeline 形式との互換のため予約) |
| 引用ブロック | anchor がある場合必須 | メタデータ行群の後、空行を挟んで `> ` で始まる blockquote。ハイライトした原文 |
| 本文 | 任意 | blockquote の後、空行を挟んで自由記述 |

**パース規則:** エントリ先頭から「タイムスタンプ行 → `key: value` 行の連続 → 空行」をメタデータとして読む。最初の空行以降はすべて本文(blockquote 含む)。この形式に合致しないエントリも「本文のみのエントリ」として表示は継続する(壊さない・落とさない)。

- gemihub の Timeline パーサはメタデータ行の順序固定(`id:` の直後の `pinned:` のみ認識)だが、本仕様は上記の汎用 `key: value` パースとする(上位互換)。移植時は `parsePostBlock` の拡張が必要。
- `anchor:` が無いエントリでは、本文先頭が blockquote であってもそれは「引用」ではなく通常の本文として扱う(引用としての特別表示・ジャンプ対象になるのは anchor 付きエントリのみ)。

## 6. アンカー仕様

ファイルタイプごとに使うアンカーが異なる。共通原則: **引用文字列(blockquote)を第一のアンカーとし、位置情報は補助**とする。これによりドキュメントの軽微な変更・EPUB のリフローに耐える。

**照合の空白正規化(全ファイルタイプ共通・必須):** レンダリング後のテキストはノード分割・改行位置が原文と一致しないため、検索対象テキストと引用文字列の**双方**を空白正規化(改行を含む連続空白文字を半角スペース 1 つに畳み、前後をトリム)してから照合する。複数行引用(`> ` が複数行)は各行の `> ` を除去した上で半角スペースで結合し、1 つの検索文字列とする。quote-prefix / quote-suffix の照合も同様。

### 6.1 PDF

**前提: pdf.js への移行。** 現行の PDF 表示はネイティブビューアの iframe 埋め込み(`PdfDocumentFrame`)であり、テキストレイヤー・選択イベント・現在ページの取得・指定ページへのジャンプのいずれも不可能。本機能の実装に先立ち、PDF 表示を pdf.js ベースに置き換える:

- ページ単位のレンダリング(canvas)+テキストレイヤーの重畳
- 既存のフォントスケール/表示幅設定に相当するズーム
- 現在ページの取得、指定ページへのジャンプ
- テキストレイヤー上の選択イベント取得(§7.2 の選択フローに使用)

アンカー仕様:

- `anchor: page=<ページ番号>`(1 始まり)
- 位置特定: 指定ページのテキストレイヤーから引用文字列を検索(空白正規化照合)してハイライト。見つからなければページ先頭へのジャンプのみ行い、ハイライトは省略。

### 6.2 EPUB

- `anchor: spine=<spine インデックス>`(0 始まり)
- 位置特定: 該当 spine 内で引用文字列(+prefix/suffix)を検索。フォントサイズ・余白変更(リフロー)後も引用文字列検索で位置を再解決できること。
- 引用文字列が見つからない場合は spine 先頭へジャンプ。

### 6.3 Markdown / プレーンテキスト / HTML

- `anchor: text`(位置補助情報なし。引用文字列のみで特定)
- 位置特定: レンダリング後のテキストコンテンツから引用文字列を検索(空白正規化照合)。複数一致時は prefix/suffix で絞り込み、それでも複数なら最初の一致を採用。
- ドキュメントが編集されて引用文字列が消失した場合: ハイライト表示せず、タイムライン側のエントリに「リンク切れ」インジケータ(例: グレーのリンクアイコン)を表示。エントリ自体は消さない。

### 6.4 画像ファイル

アンカー非対応。通常メモ(anchor なしエントリ)のみ可。

## 7. UI 仕様

### 7.1 タイムラインパネル

- ウィジェットヘッダーにメモアイコンを追加。クリックでウィジェットの**左側**にタイムラインパネルを開閉する(添付イメージのレイアウト)。
- パネルはウィジェット単位。同時に複数ウィジェットのパネルを開ける。
- パネルヘッダーの「«」でパネルを細いレール状に折りたたみ、レールの「»」で再展開できる(状態はウィジェット設定として永続化)。
- 表示は**古い順**(ファイルと同順。最新エントリが下部のコンポーザー直上に来る。エントリ追加時は最下部へ自動スクロール)。
- 各エントリの表示: タイムスタンプ、id、引用(あれば)、本文。長い本文は折りたたみ「もっと見る」。
- エントリ操作: 編集・削除・ピン留め(既存 Timeline UI があればそれに準拠)。編集・削除はファイル内の該当エントリを id で特定して書き換える。
- パネル下部に投稿コンポーザー(WYSIWYG / Raw 切替可)。投稿でファイル末尾に追記。
- メモファイルが外部エディタ等で変更された場合の自動反映は考慮しない(§10 スコープ外)。

### 7.2 アンカー付きエントリの作成フロー

1. ドキュメント(プレビュー / PDF / EPUB 表示)内でテキストを選択
2. 選択ポップアップまたはコンテキストメニューに「メモを追加」を表示
3. クリックでタイムラインパネルを開き(閉じていれば)、コンポーザーに以下をプリセット:
   - anchor(ファイルタイプに応じて自動生成)
   - 引用 blockquote(選択テキスト)
   - quote-prefix / quote-suffix(選択範囲の前後 30 文字を自動取得。取得不能なら省略)
4. ユーザーが本文を書いて投稿 → ファイル末尾に追記

選択なしでの通常投稿(anchor なし)も常に可能。

### 7.3 ドキュメント側のハイライト表示

- ハイライトはパネルが開いている間(「«」で折りたたんだレール状態を含む)適用し、「×」で閉じたときのみドキュメントを素の表示に戻す。折りたたみ中にハイライトをクリックするとパネルが再展開される。
- パネル表示中は、メモファイルを読み、現在表示中の範囲に解決できるアンカー付きエントリがあれば、該当テキストの**文字色を変更**する(下線等ではなく色。テーマごとに識別しやすいアクセント色を定義し、light/dark 両テーマで可読性を保つ)。
- 実装は **CSS Custom Highlight API(`::highlight()`)を推奨**。DOM を書き換えずに複数テキストノードにまたがる Range へ色を適用でき、プレビューの再レンダリングと干渉しない。WebView2(Chromium)/WebKit とも対応済み。
- 同一箇所に複数エントリがある場合は 1 つのハイライトにまとめる。
- **ホバー**: ハイライト部分にマウスオーバーでポップオーバーを表示。内容はメモ本文の先頭(最大 200 文字程度、超過は省略記号)。複数エントリ時は件数と最新エントリの冒頭。
- **クリック**: タイムラインパネルを開き(閉じていれば)、該当エントリまでスクロールし、エントリ背景を短時間フラッシュ(1 秒程度のフェード)して視線誘導する。複数エントリ時は最新のものへ。

### 7.4 タイムライン → ドキュメントへのジャンプ

- アンカー付きエントリの引用部分(またはジャンプアイコン)をクリックすると、ドキュメント側を該当位置までスクロールし、該当テキストを短時間フラッシュする。
- PDF/EPUB では該当ページ/spine への移動を伴う。
- 解決不能(リンク切れ)の場合はトースト等で「元の位置が見つかりません」と通知し、何もしない。

### 7.5 wikiリンク

- メモ本文内の `[[...]]` は既存の wikiリンク解決に従う。埋め込み `![[...]]`(画像等)も同様。
- 解決の基準ディレクトリはメモディレクトリとする。

## 8. 動作詳細・エッジケース

1. **書き込みの安全性**: 投稿は末尾追記のみ。編集・削除時のみ全体書き換えを許可し、書き込みは一時ファイル+rename のアトミック書き込みとする。
2. **ドキュメント移動**: メモファイルが見つからない場合は単に「メモなし」として扱う(新規投稿で新しい名前のファイルが作られる)。ユーザーが手動 rename で復旧する運用を前提とし、アプリ側での自動追跡はしない(v1 スコープ外)。
3. **不正なメモファイル**: パースに失敗した部分は本文のみのエントリとして表示し、エラーで全体を落とさない。ユーザー編集(外部エディタ)を前提とする。
4. **パフォーマンス**: ハイライト解決(引用文字列検索)はドキュメント表示時と表示範囲変更時(PDF ページ送り、EPUB spine 移動)に行う。Markdown ではプレビュー再レンダリング時。全文検索は表示中の範囲に限定してよい。
5. **id 衝突**: 投稿時にファイル内の既存 id 一覧と照合し、衝突する場合は `-2`, `-3`… のサフィックスを付与して一意化する(gemihub Timeline の `uniquePostId` と同方式)。ミリ秒繰り上げは id が実時刻と乖離するため採用しない。
6. **同一ドキュメントを複数ウィジェットで表示中の編集・削除**: 全体書き換えの直前にファイルをディスクから再読込し、対象エントリを id で特定して書き換える。メモリ上の古い内容から書き戻すと、他パネルからの投稿が消失するため。

## 9. 必要なバックエンド API(新設)

現行の Go バックエンド(`app.go`)は**読み取り専用**(ファイル選択ダイアログとファイル読み取りのみ)のため、以下を新設する。

| API | 用途 |
|---|---|
| ディレクトリ選択ダイアログ | メモディレクトリ設定(§3)。現在はファイルダイアログのみ |
| メモファイルの読み取り・存在確認 | パネル表示・解決順序(§4.3) |
| 末尾追記 | 投稿(§8.1) |
| アトミック全体書き換え(一時ファイル+rename) | 編集・削除(§8.1) |

## 10. スコープ外(v1 では実装しない)

- メモファイルの外部エディタ編集の自動反映(ファイル監視)
- ドキュメント移動の自動追跡(内容ハッシュによる再関連付け)
- メモの全ドキュメント横断ビュー
- WYSIWYG モードでの Markdown 編集中のハイライト表示(プレビューモードのみ対応)
- タグによるフィルタリング

## 11. 受け入れ基準

1. パスエンコードの往復変換テストが通る(`_`/`/`/`\`/`:` を含むパス、日本語パス、200 バイト超フォールバック)
2. メモディレクトリ設定 → md/pdf/epub 各ドキュメントでメモアイコン → パネル開閉ができる
3. テキスト選択 → メモ追加 → ファイル末尾に正しい形式で追記される
4. 追記されたエントリの引用箇所がドキュメント上で色変わりし、ホバーで本文プレビュー、クリックでタイムラインの該当エントリへスクロール+フラッシュする
5. タイムライン側の引用クリックでドキュメントが該当位置へスクロール+フラッシュする
6. EPUB でフォントサイズ・余白を変更した後もハイライトとジャンプが機能する
7. ドキュメント編集で引用文が消えた場合、リンク切れ表示になりアプリは落ちない
8. PDF が pdf.js で表示され(ズーム・ページ移動含む)、テキスト選択からのメモ追加、ページジャンプ、ハイライトが機能する
9. frontmatter(`source:`)がエントリとして誤表示されない。本文中に `---` を含むコードブロックがあるエントリが分断されない
